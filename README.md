# svtracker

Shadowverse: Worlds Beyond (Steam版) の対戦画面を監視し、

- 公式サイトのカード画像と画面上のカードを照合してプレイされたカードを判定
- ターン数・自分と相手の行動を記録
- 記録した対戦データから相手の次の行動を予測
- 現在の盤面・手札から最善に近い行動を提案

することを目指す個人利用向けツールです。

## できること / できないこと

- **できる**: カード画像同士の照合(pHash)、手札→盤面の差分によるプレイ検出、
  OCR/ピクセル色判定によるターン数・手番・PP・ライフ・エクストラPP・進化ポイント(EP)の読み取り、
  認識したカードのクラス情報からの自分/相手クラス自動判別(手入力不要、デッキが単一クラス固定
  であることを利用してニュートラル以外のカードが1枚でも認識できれば判定)。
  ゲーム内のバトルログに近い形で、カードのプレイ・進化/超進化に加えて、盤面からの
  ユニット消失(破壊/除去など、原因までは区別しません)・ライフ増減・ターン終了も
  検出してSQLiteに記録します(いずれも画面認識のフレーム間差分による検出のため、
  認識漏れ・誤検出はありえます)。
  記録に基づく簡易な相手行動予測、盤面状況からの(自分視点・相手視点双方の)リーサル検出・
  トレード提案・エクストラPPを含めたPP消費提案・進化ポイントを使うべきタイミングの提案。
  自分がリーサルを取れる場面だけでなく、相手の盤面の合計攻撃力が自分のライフ以上になっている
  場合は「相手のリーサルに警戒」として最優先で警告し、ブロッカー温存・除去/回復の優先など
  リーサルケアを意識した立ち回りを促します。
  進化/超進化そのものも、進化ポイントが減ったフレームを検知してアクション履歴に記録します
  (誰が・何ターン目に進化したか。どのユニットに進化したかまでは特定しません、後述の制限を参照)。
  対戦記録が十分あれば、進化ポイントが残っている場面で「そのクラス相手に過去進化した対戦の
  勝率」も併せて表示します。
  対戦形式(アンリミテッド/ローテーション)も監視タブから選択でき、ローテーション選択時は
  相手プレイ予測のカードプールをローテーション対象カードのみに絞り込めます
  (要 `rotation_min_card_set_id` 設定、後述の制限を参照)。
  監視終了時に対戦の勝敗(勝ち/負け/記録しない)を記録でき(画面から自動判定はしないため
  手動確認)、蓄積した勝敗データから「そのクラス相手にプレイして過去の勝率が高かった
  手札のカード」を優先プレイ候補として提案します(対戦数が少なすぎる場合は提案しません)。
  この勝率データはPP消費提案の組み合わせ選びにも使われ、単にPPを使い切る組み合わせでは
  なく、過去の勝率が高いカードを優先的に含む組み合わせを提示します(対戦記録が十分無い
  場合は従来どおりPP消費量の最大化のみで選びます)。
  カードにPP上限増加・PP回復・進化ポイント回復の効果量を設定しておけば(手動入力、後述)、
  PP消費提案がPP回復分を加味した組み合わせを提示し、PP上限を増やせるカードや進化ポイントを
  回復できるカードは優先プレイ候補として別途提示されます。
  これらはシェル操作なしで完結するデスクトップGUI(Tkinter、`svtracker-gui`)からも操作できます。
- **精度に限界がある**: 盤面ユニットのATK/HPはカードマスタの基礎値をそのまま使っており、
  バフ/デバフ・疲労状態(攻撃済みかどうか)・進化による能力値上昇は画面から読み取っていないため、
  盤面表示は常に「攻撃可能・未進化」扱いです(相手のリーサル警戒も同じ前提のため、実際には
  疲労中で攻撃できないユニットを含めて警戒してしまう=安全側に多めに警告する場合があります)。
  進化イベント自体は進化ポイントの増減で検出できますが、
  盤面の複数ユニットの中でどれが進化したかまでは特定していません。より正確にするにはユニット上の
  ATK/HP表示OCRや、盤面枠ごとの進化演出(枠色の変化など)検出の追加実装が必要です。
  盤面からのユニット消失検出も同様にフレーム間の単純差分のため、認識が1フレームだけ
  失敗して復帰した場合に「消失」と「再出現」を誤って記録することがあります
  (手札/盤面のプレイ検出も同じ弱点を元々持っています)。
  ローテーション絞り込みは各カードの `card_set_id`(カードセット/弾番号)としきい値
  `rotation_min_card_set_id` の比較で行っていますが、`card_set_id` を取得元(公式サイトAPI)の
  どのフィールドから読むかは未検証の推測です(`src/svtracker/cards/card_fetcher.py` の
  `_parse_card_common` 内コメント参照)。実際のレスポンスで名前が違えば全カードが
  `card_set_id=None` のままになり、絞り込みは効かず実質アンリミテッドと同じ結果になります
  (安全側のフォールバックであり、誤って対象カードを除外することはありません)。また
  ローテーションの区切り(しきい値)はゲーム側の改定で変わるため、`config/settings.json` の
  `rotation_min_card_set_id` を都度更新する必要があります。
  PP上限増加/PP回復/進化ポイント回復の効果量は、カードの能力テキスト由来のデータであり
  公式サイトAPIには構造化された形で含まれていないとみられるため、`fetch-cards` では
  常に「効果なし(0)」として取得されます。これらの提案を使うには `import-cards` のCSVで
  該当カードの効果量を手動指定してください(後述)。
  トークンカード(効果で生成されるカード)は公式サイトのカード一覧に含まれないとみられるため
  `fetch-cards` だけでは取得できません。`import-cards --merge` で画像+CSVを別途追加すれば、
  一度DBに入った後は通常のカードと同様に認識・予測・アドバイスの対象になります(後述)。
- **できない/未検証**: 本開発環境はGUIもSteamクライアントも無いサンドボックスのため、
  実際のゲーム画面に対する認識精度・キャリブレーションはこの環境では検証できていません。
  実機(Windows + Steam版SVWB)でのキャリブレーションと調整が必須です。
  同様に公式サイト(shadowverse-wb.com)からのカード自動取得(`fetch-cards`)も、
  この開発環境からは対象サイトへのアクセスが行えなかったため実サイトに対しては未検証です。
  サイトのHTML構造が変わっている場合は `src/svtracker/cards/card_fetcher.py`
  冒頭のCSSセレクタ定数を実際のページに合わせて調整してください。
  実機での実行で、画像(`/uploads/card_image/...`)のダウンロードが全件
  `403 Forbidden` になる問題がありましたが、原因はホットリンク対策ではなく
  画像URLの言語セグメントの誤り(`ja` → 正しくは `jpn`。実機ブラウザのDevToolsで
  確認)で、修正済みです。存在しないパスに対してCDNが404ではなく403を返すため、
  アクセス制限と誤認しやすい点に注意してください。もし将来また403が全件発生する
  場合は(1回だけまとめ警告ログが出ます)、まず実機ブラウザの開発者ツールで
  成功する画像リクエストのURL・ヘッダーを確認して `CARD_IMAGE_URL_TMPL` /
  `IMAGE_REQUEST_HEADERS` / `_download_card_image` を実際のリクエストに合わせて
  調整するか、`import-cards` (後述)でローカルに用意した画像+CSVからカードDBを
  構築する方法を使ってください。なお `fetch_from_official_site` はAPIを叩く前に
  一度カード一覧ページへGETしてセッションCookieを確立します(ブラウザの
  「まずページを開く」動作の模倣。現状のCDNはCookie不要と確認できていますが、
  無害なので念のため残しています)。
  GUIについては、ウィジェットの生成・操作・保存までの一連の流れをXvfb(仮想ディスプレイ)上で
  自動チェック済みですが、実際のWindowsデスクトップでの見た目・操作感は未確認です。

## セットアップ

```bash
python3 -m venv .venv
source .venv/bin/activate   # Windowsは .venv\Scripts\activate
pip install -e ".[dev,windows,ocr]"
```

- `windows` extra: ウィンドウ自動検出 (`pygetwindow`)。Windows環境でのみ有効。
- `ocr` extra: ターン数/PP/ライフの文字をOCRで読み取るために**実質必須** (`pytesseract`)。
  別途 [Tesseract OCR本体](https://github.com/tesseract-ocr/tesseract) のインストールが必要です
  (Windowsはインストーラでパスを通すか `pytesseract.pytesseract.tesseract_cmd` を設定)。
  未導入の場合はターン/PP/ライフの自動読み取りが無効になり、ターン数は1のまま進みません。
  `pytesseract` パッケージ自体が無い場合・Tesseract本体が未インストール/PATH未設定の場合の
  いずれも、ログには初回の1回だけ警告を出します(認識のたび=毎フレーム発生しうるため、
  同じ警告を繰り返し出力することはありません)。

## 使い方(GUI版・おすすめ)

Windowsインストーラを使う場合、コマンド操作は不要です。インストール後にデスクトップ/
スタートメニューの「svtracker」を起動すると、以下の4タブを持つウィンドウが開きます。

1. **カードDB タブ**: 「公式サイトから取得」または「ローカルの画像+CSVから取り込み」ボタンで
   カードマスタを用意します。公式サイトに載らないトークンカードを追加したい場合は、
   「既存のカードDBに追加する(上書きしない)」にチェックを入れてから「ローカルから取り込み」で
   トークン用の画像+CSVを取り込むと、公式データを消さずに追加できます。
2. **キャリブレーション タブ**: 上部でキャプチャするモニタを選び、「スクリーンショットを撮る」
   ボタンで画面を取得します。右側の一覧から編集したい領域(手札・盤面・ターン表示など)を
   選んでから、画面プレビュー上をドラッグ(矩形)またはクリック(座標・手番判定用ピクセル)で
   指定します。手札・盤面のような複数枠の領域は、先頭2〜3枚だけドラッグして枚数を指定し
   「等間隔で自動補完」を押せば、間隔を推定して残りを自動生成できます(全部手でドラッグ
   する必要はありません)。手番判定の基準色も、プレビュー上をクリックするだけで登録できます。
   「regions.json / settings.json に保存」で確定します。
3. **監視 タブ**: 自分/相手のクラス欄は空欄のまま「開始」して構いません。手札・盤面で
   最初に認識できたニュートラル以外のカードから自動でクラスを判別し(デッキは単一クラス
   固定なので1枚判別できれば十分)、判別結果はログ欄と欄下の表示に反映されます。
   誤認識対策のため一度判明したクラスは上書きしません。認識に頼らず固定したい場合は
   従来どおり手入力でも指定できます(手入力があればそちらを優先し、自動判別しません)。
   「対戦形式」ドロップダウンでアンリミテッド/ローテーションを選べます(選択は
   `config/settings.json` に保存され次回起動時も維持されます)。ローテーションを選んだ場合、
   相手プレイ予測の候補カードをローテーション対象のみに絞り込みます(絞り込みには
   `rotation_min_card_set_id` の設定が必要。前述の「できること/できないこと」を参照)。
   認識したプレイ・予測・提案はログ欄にリアルタイム表示されます。「停止」を押すと
   対戦結果を確認するダイアログが出るので「勝ち」「負け」「記録しない」から選べます
   (画面から自動判定はしていません)。ここで記録した勝敗が、統計タブの勝率表示や
   後述の「勝率の高いカード」提案の元データになります。
4. **統計 タブ**: 相手クラス名を入力すると、その相手との戦績(勝敗・勝率)と、
   そのクラス相手に過去よく使われたカードの一覧が表示されます。

GUIも内部的には下記CLIと同じロジック(`svtracker.app.MonitorApp` など)を呼んでいるだけなので、
挙動や設定ファイルの場所は完全に共通です。

### コマンドラインが必要な場合

自動化やデバッグのために、GUIとは別に `svtracker`(CLI, コンソール表示あり)コマンドも
同梱されています。以下は主にそちらの説明です。

## 使い方(CLIコマンド版)

### 1. カードマスタDBを用意する

公式サイトから自動取得を試す場合:

```bash
svtracker fetch-cards
```

うまく取得できない場合は、手元に用意した画像+CSVから取り込みます:

```bash
svtracker import-cards ./my_card_images ./my_card_metadata.csv
```

CSVフォーマット (`data/cards/` 配下の画像を想定):

```csv
card_id,name,clan,cost,card_type,rarity,filename,base_atk,base_hp,card_set_id,max_pp_boost,pp_recover,ep_recover
10112210,ロビンフッド,エルフ,4,フォロワー,レジェンド,10112210.png,3,3,10,0,0,0
```

`card_set_id` はローテーション絞り込みに使うカードセット(弾)番号で省略可能です
(省略した場合、そのカードはローテーション絞り込みで除外されません)。
`max_pp_boost`/`pp_recover`/`ep_recover` はそれぞれ「プレイすると永続的にPP上限を+N」
「そのターンのPPを+N回復」「進化ポイントを+N回復」する効果を持つカードの効果量です
(すべて省略可、既定0=効果なし)。`fetch-cards` ではこれらの効果データは取得できないため、
既に取得済みのカードDBに後から効果量だけ補いたい場合は、`images_dir` に
`fetch-cards` で保存済みの `data/cards/` を指定し、`filename` に既存の画像ファイル名
(`{card_id}.png`)を指定した上で `import-cards --merge` を使ってください。

#### トークンカードを追加する

公式サイトのカード一覧には、効果で生成される「トークンカード」(手札やデッキに
直接入れられないカード)が含まれていないとみられます。トークンも認識・予測・提案の
対象にしたい場合は、画像+CSVを別途用意して `--merge` 付きで取り込むと、
既存のカードDBを消さずに追加できます。

```bash
svtracker fetch-cards                                     # まず公式カードを取得
svtracker import-cards ./token_images ./tokens.csv --merge  # トークンを追加(公式データは残る)
```

`--merge` を付けなかった場合、そのコマンドで取得/取り込んだ内容でカードDB全体が
上書きされる点に注意してください(`fetch-cards --merge` も同様に使えます)。
一度カードDBに入ってしまえば、トークンかどうかで扱いを区別する仕組みは無いので、
通常のカードと同じように認識・盤面反映・予測・アドバイスの対象になります。

### 2. 画面領域をキャリブレーションする

複数モニタ環境では、まずキャプチャ対象のモニタを確認してください。

```bash
svtracker list-monitors
```

ゲームを起動した状態で1枚キャプチャを保存し、手札・盤面の座標を調べます。

```bash
svtracker screenshot -o screenshot.png
# 特定のモニタを指定する場合:
svtracker screenshot -o screenshot.png --monitor 2
```

`--monitor` を省略した場合は `config/settings.json` の `monitor_index` が使われます。
GUI版では「キャリブレーション」タブ上部のモニタ選択欄で切り替えられ、選択は即座に
`config/settings.json` に保存されます。

`config/regions.example.json` を `config/regions.json` としてコピーし、
保存した画像を見ながら各カード枠の `[x, y, width, height]` を実際の座標に書き換えてください。
手札・盤面のような複数枠の領域を1枠ずつ手打ちするのは大変なので、GUI版の「等間隔で
自動補完」機能(先頭2〜3枚の座標から間隔を推定して残りを生成)を使うことをおすすめします。

ターン数・PP・ライフ・エクストラPP・進化ポイントの自動読み取りには `turn_indicator` /
`self_pp` / `self_life` / `opponent_life` / `self_extra_pp` / `self_ep` / `opponent_ep` の
各領域も実際の数字表示部分に合わせてください(OCRなので、余白を含めすぎず数字部分だけを
囲むと精度が上がります)。`self_extra_pp` はPP表示の近くにある持ち越し分(0〜2)、
`self_ep`/`opponent_ep` は残り進化ポイント表示です。進化ポイントが数字ではなく
アイコン(宝石など)で表示されるUIの場合はOCRでは読み取れないため、その場合は
これらの領域を未設定のままにしてください(進化検出・進化ポイント関連の提案のみ
無効になり、他の機能には影響しません)。

手番判定 (`active_player_pixel`) は数字OCRではなくピクセル色判定です。自分の手番の時と
相手の手番の時、それぞれのスクリーンショットで手番表示の色が変わるUIパーツ(背景色や
ハイライトなど)を1点選び、その座標を `active_player_pixel` に、実際の色(RGB)を
`config/settings.json` の `self_turn_color` / `opponent_turn_color` に設定してください。
`config/settings.example.json` の値はダミーなので必ず実際の色に置き換える必要があります。

### 3. 監視を開始する

```bash
svtracker run --self-clan エルフ --opponent-clan ロイヤル --format rotation
```

一定間隔(既定1秒、`config/settings.json`の`capture_interval_sec`で変更可)で画面をキャプチャし、
検出したプレイを標準出力にログしながら、相手の次の行動予測と自分への行動提案を表示します。
`--format` は `unlimited` / `rotation` から選択(省略時は `config/settings.json` の
`game_format`)。`rotation` 選択時に相手プレイ予測の候補を絞り込むには、`config/settings.json`
の `rotation_min_card_set_id` を設定しておく必要があります。
`Ctrl+C` で終了すると対戦結果を記録するか尋ねられます
(`w`=勝ち / `l`=負け / それ以外の入力=記録しない)。ここで記録した勝敗が、
`stats` コマンドの勝率表示や「勝率の高いカード」提案の元データになります。

### 4. 蓄積した対戦統計を見る

```bash
svtracker stats ロイヤル
svtracker stats ロイヤル --self-clan エルフ  # 自分のクラスでも絞り込む場合
```

相手クラスに対する戦績(勝敗数・勝率)と、そのクラス相手によく使われたカードの
一覧が表示されます。

## 設定ファイル

- `config/settings.json` (`config/settings.example.json` をコピーして作成): キャプチャ間隔、
  モニタ番号、マッチング閾値、手番判定用の基準色 (`self_turn_color` / `opponent_turn_color`)、
  対戦形式 (`game_format`: `"unlimited"` / `"rotation"`) とローテーション絞り込みしきい値
  (`rotation_min_card_set_id`) など。
- `config/regions.json` (`config/regions.example.json` をコピーして作成): 手札・盤面などの
  画面上の矩形領域と、手番判定用の座標 (`active_player_pixel`)。
  **解像度・UIレイアウトごとに調整が必須**です。

## Windowsインストーラ

Python環境を用意しなくても使えるように、PyInstallerでの単体exe化(GUI版・CLI版の両方)と
Inno Setupでのインストーラ作成に対応しています。実際のビルドはGitHub Actions
(`windows-latest`ランナー)で行われるため、この開発環境(Linuxサンドボックス)では
作れませんが、CIで実機Windows相当の環境を使ってビルド・GUI起動・動作確認しています。

- 入手方法: GitHubリポジトリの Actions タブ → `Build Windows Installer` ワークフロー
  → 対象の実行 → Artifacts から `svtracker-windows-installer` (インストーラ) または
  `svtracker-exe` (GUI版 `svtracker-gui.exe` とCLI版 `svtracker.exe` の単体exe2つ) を
  ダウンロード。タグ `v*` をpushすると自動でも走ります。
- インストーラは `svtracker-gui.exe`(GUI・デスクトップ/スタートメニューのメインアイコン)、
  `svtracker.exe`(CLI・上級者向けの別ショートカット)、`config/*.example.json`、READMEを
  `Program Files\svtracker` 配下に配置します。`data/`, `config/settings.json`,
  `config/regions.json` はGUIの「キャリブレーション」タブから作成・保存できます。
- **OCR(ターン/PP/ライフ読み取り)を使うには、インストーラとは別に
  [Tesseract OCR本体](https://github.com/tesseract-ocr/tesseract) のインストールが必要です**
  (バイナリはライセンス・サイズの都合でインストーラに同梱していません)。
- ローカルでビルドする場合は `packaging/entrypoint.py`(CLI)・
  `packaging/gui_entrypoint.py`(GUI)をそれぞれ PyInstaller で、
  `packaging/installer.iss` を Inno Setup (`ISCC.exe`) でビルドしてください。
  具体的なコマンドは `.github/workflows/build-windows-installer.yml` を参照してください。

## 仕組みの概要

```
画面キャプチャ(mss)
   -> 領域切り出し(config/regions.json)
   -> ターン数/手番/PP/ライフ/エクストラPP/進化ポイントを読み取り
      (capture/ocr_reader.py: OCR + ピクセル色判定)
   -> カード照合(pHashで公式カード画像DBと比較, cards/card_matcher.py)
   -> 手札/盤面の差分からプレイ・盤面からのユニット消失を検出、進化ポイントの減少から
      進化/超進化を検出、ライフ増減・ターン終了も検出(ゲーム内バトルログ相当の記録。
      game/event_detector.py)
   -> 対戦状態を記録(game/match_tracker.py) + SQLiteに永続化(storage/match_log.py)
   -> 監視終了時に勝敗を記録(GUIのダイアログ/CLIのプロンプト。画面からの自動判定はしない)
   -> 相手の次の一手を予測(prediction/predictor.py: 履歴頻度 + 新規性 + PPカーブ適合度)
   -> 自分の最善に近い行動を提案
      (prediction/advisor.py: 自分/相手両視点のリーサル判定(相手視点はリーサルケア警告)/
      トレード判定/過去の勝率データで重み付けしたPP消費最適化(エクストラPP・PP回復カード
      も考慮)/過去の進化時勝率を添えた進化ポイント活用・回復提案/PP上限増加カードの優先提案/
      過去の勝敗記録に基づく勝率の高いカードの提示)
```

予測・アドバイスは厳密なゲームAI/探索ではなく、説明可能なヒューリスティックです。
対戦記録が蓄積されるほど「相手の次の行動予測」・「勝率の高いカード」提案の精度は
上がっていきます(初回や対戦記録が無いクラス相手では、相手行動予測はカーブ適合度と
新規性のみで行い、勝率に基づく提案は対戦数が一定数貯まるまで表示されません)。

## テスト

画面キャプチャ・公式サイト通信を含まない部分(カード照合・対戦記録・予測・アドバイス)は
オフラインでテストできます。

```bash
pip install -e ".[dev]"
pytest
```

## 注意事項

- 本ツールは画面をキャプチャして解析するだけで、ゲームクライアントへの
  自動操作・メモリ改ざん・通信の書き換えなどは一切行いません。
- 公式サイトのカード画像を取得・保存する機能を含みます。利用は個人の学習・観戦補助目的の
  範囲にとどめ、Cygames社の利用規約を確認の上で自己責任で使用してください。
- 対戦相手の情報を記録する機能はローカルのSQLiteファイルに保存するのみで、
  外部送信は行いません。
